What Are Micro Frontends?
Micro frontends extend the idea of microservices to the frontend. Instead of one large, tightly coupled frontend app, you split it into smaller, independently developed and deployed pieces—each owned by a team.

Well-known platforms like Spotify, IKEA, and Zalando have used micro frontend principles to scale large web applications across teams.

✅ Benefits of Micro Frontends
1. Independent Teams
Each team owns a slice of the UI (e.g., checkout, search, profile).

Faster development cycles

Clear ownership

Less cross-team blocking

2. Independent Deployments
You can deploy one micro frontend without redeploying the entire app.

Reduced release risk

Faster feature rollouts

Easier rollbacks

3. Technology Flexibility
Different teams can use different frameworks if needed:

React for one module

Vue for another

Angular for legacy parts

This is especially useful during gradual migrations.

4. Incremental Upgrades
You can modernize parts of a monolith without rewriting everything.

Replace one section at a time

Lower migration risk

5. Better Scalability (Org + Codebase)
As organizations grow:

Codebases stay manageable

Cognitive load per team decreases

Onboarding becomes easier

6. Fault Isolation
If one micro frontend fails, the rest of the app can still function (if properly isolated).

⚠️ Trade-Offs (Important to Know)
Micro frontends are not free:

More infrastructure complexity

Shared dependency duplication risk

Cross-app communication challenges

Harder consistency (design systems, state management)

They’re best suited for:

Large teams (multiple squads)

Large, long-lived products

Independent domain areas

Not ideal for:

Small apps

Small teams

Early-stage startups

Step 1: Evaluate If You Really Need Them

Combine apps during build.

Monorepos (e.g., Nx)

Simpler setup

Best for starting.

2️⃣ Runtime Integration

Common tools:

3️⃣ Web Components
Use browser-native custom elements.

Step 4: Start Small (Strangler Pattern)
Pick one isolated feature and extract it.

For example:

Step 5: Establish Shared Foundations

🔢 3️⃣ Versioning Strategy (Very Important)
This is where many micro frontend projects fail.

There are 3 layers to version:

A) Deployment Versioning
Each micro frontend should:

Be independently deployable

Have semantic versioning (semver)

Example:

product-app v1.4.2
cart-app v2.1.0
B) Shared Dependency Versioning
🔴 Biggest Risk: React Duplication
If product uses React 18.2 and cart uses 18.0 → boom.

Example:

  quantity: number
}

Solution:

Shared contract package (@company/contracts)

Strict versioning

Avoid implicit assumptions

Webpack Module Federation — What It Is (Simply Explained)
Webpack Module Federation (MF) is a feature introduced in Webpack 5 that allows separate applications to share code at runtime.

In micro frontends, it lets you:

Load parts of other apps dynamically

Share dependencies (like React) without duplication

Deploy apps independently

Compose multiple frontends into one UI

It’s one of the most common tools used to implement micro frontends today.

🧠 Core Idea
Instead of bundling everything into one big file at build time, Module Federation allows:

“Load that component from another app — at runtime.”

So your shell app can load components from a remote app hosted somewhere else.

🏗 Basic Architecture
You typically have:

Shell (host)
   ↓
Loads
   ↓
Remote apps (micro frontends)
Example:

Shell = container app

Product app = remote

Cart app = remote

🔌 Key Concepts
1️⃣ Host
The app that loads other apps.

2️⃣ Remote
An app that exposes modules to be consumed.

3️⃣ Exposes
What the remote makes available.

4️⃣ Shared
Dependencies shared between host and remotes.

🛠 Basic Example
Remote App (product)
// webpack.config.js
new ModuleFederationPlugin({
  name: "product",
  filename: "remoteEntry.js",
  exposes: {
    "./ProductApp": "./src/ProductApp",
  },
  shared: {
    react: { singleton: true },
    "react-dom": { singleton: true }
  }
})
This creates:

remoteEntry.js
That file is what other apps load.

Host App (shell)
new ModuleFederationPlugin({
  name: "shell",
  remotes: {
    product: "product@https://cdn.company.com/product/remoteEntry.js"
  },
  shared: {
    react: { singleton: true },
    "react-dom": { singleton: true }
  }
})
Then you can import dynamically:

const ProductApp = React.lazy(() => import("product/ProductApp"));
Now it loads at runtime.

🚀 Why It’s Powerful
✅ Independent deployment
Deploy product app without touching shell.

✅ Shared dependencies
Avoid duplicate React versions.

✅ Runtime composition
Apps can be loaded dynamically from a CDN.

✅ Gradual migration
Extract parts of monolith step by step.

🧩 What Happens at Runtime?
When user visits:

Shell loads

Shell fetches remoteEntry.js

Webpack negotiates shared dependencies

Remote component loads

UI renders

All in the browser.

🔥 Real-World Setup
CDN
 ├── shell/
 ├── product/
 ├── cart/
Shell loads:

product@https://cdn.company.com/product/remoteEntry.js
Each app versioned independently.

⚠️ Important Considerations
1️⃣ Dependency Sharing Is Critical
If you forget:

shared: { react: { singleton: true } }
You may load multiple React instances → crashes.

2️⃣ Version Compatibility
If host uses React 18.2 and remote uses 17 → conflict.

Best practice:

Align major versions

Use strict semver

CI validation between apps

3️⃣ Performance
Each remote adds:

Extra network request

Extra JS bundle

Too many micro frontends → slower startup.

🆚 Module Federation vs Alternatives
Approach	Pros	Cons
Module Federation	Runtime flexibility	More complexity
Monorepo shared libs	Simpler	No independent deployment
iFrames	Strong isolation	Bad UX integration
Web Components	Framework-agnostic	Limited state sharing

Large platforms and enterprises adopt MF to scale teams. It’s widely used across modern React ecosystems.

🧠 When You Should Use It
Use Module Federation if:

Large evolving product

Plugin/extension architecture

Avoid if:

Early startup
